# Database

## PostgreSQL role

PostgreSQL is the system of record for users, knowledge bases, document
ingestion metadata, extracted text chunks, conversations, and chat messages.
Qdrant stores derived vectors for retrieval, while PostgreSQL owns tenant
relationships, document order, and conversation history.

The application uses PostgreSQL 17 through SQLAlchemy's asynchronous engine and
the `asyncpg` driver. Local Docker Compose exposes the database on port `5432`.

## Migrations

Alembic exclusively owns relational schema creation and evolution. FastAPI
startup does not call `create_all` and never mutates the schema implicitly.

Apply all pending migrations before starting a new environment:

```bash
alembic upgrade head
```

Display the currently applied revision:

```bash
alembic current
```

After changing SQLAlchemy models, generate a candidate migration:

```bash
alembic revision --autogenerate -m "describe schema change"
```

Review every autogenerated migration before applying it. Alembic compares the
metadata registered through `app.models` with the live PostgreSQL schema; model
imports in `migrations/env.py` ensure that every table is registered.

Rollback the latest revision during development with:

```bash
alembic downgrade -1
```

The migration history starts with
`migrations/versions/20260704_0001_initial_schema.py`. Multi-user ownership is
introduced by `20260704_0002_multi_user_foundation.py`.
Conversation memory is introduced by
`20260704_0003_conversation_memory.py`.
Asynchronous document state is introduced by
`20260705_0004_document_processing_status.py`.

### Existing pre-migration local databases

Databases created before Alembic may already contain `documents` and
`document_chunks` but have no `alembic_version` row. Verify that their schema
matches the initial migration, baseline that exact revision, and then apply the
ownership migration:

```bash
alembic stamp 20260704_0001
alembic upgrade head
```

The ownership migration preserves those rows under a conditional
`Legacy Documents` knowledge base. Do not use `stamp` on an empty database
because it records a revision without creating tables. Fresh databases must use
`alembic upgrade head`.

## `users` table

| Column | Type | Purpose |
| --- | --- | --- |
| `id` | UUID, primary key | Application-generated user identifier |
| `email` | VARCHAR(320), unique | Normalized account email |
| `created_at` | TIMESTAMPTZ | Database-generated creation time |

No password or authentication credential is stored in STEP 10.

## `knowledge_bases` table

| Column | Type | Purpose |
| --- | --- | --- |
| `id` | UUID, primary key | Application-generated knowledge-base ID |
| `user_id` | UUID, foreign key | Owning user |
| `name` | VARCHAR(255) | Display name |
| `description` | TEXT, nullable | Optional description |
| `created_at` | TIMESTAMPTZ | Database-generated creation time |

Deleting a user cascades to their knowledge bases.

## `conversations` table

| Column | Type | Purpose |
| --- | --- | --- |
| `id` | UUID, primary key | Application-generated conversation ID |
| `knowledge_base_id` | UUID, foreign key | Knowledge base used by the chat |
| `title` | VARCHAR(255), nullable | Optional display title |
| `created_at` | TIMESTAMPTZ | Database-generated creation time |

`knowledge_base_id` is indexed. Deleting a knowledge base cascades to its
conversations and messages.

## `messages` table

| Column | Type | Purpose |
| --- | --- | --- |
| `id` | UUID, primary key | Application-generated message ID |
| `conversation_id` | UUID, foreign key | Parent conversation |
| `role` | VARCHAR(20) | Either `user` or `assistant` |
| `content` | TEXT | Complete message content |
| `created_at` | TIMESTAMPTZ | Message ordering and creation time |

`conversation_id` is indexed and uses `ON DELETE CASCADE`. A database check
constraint rejects roles other than `user` and `assistant`. QA writes the user
question and assistant answer in one transaction after a successful answer.

## `documents` table

One row represents one accepted PDF or TXT upload.

| Column | Type | Purpose |
| --- | --- | --- |
| `id` | UUID, primary key | Application-generated document identifier |
| `knowledge_base_id` | UUID, foreign key | Owning knowledge base |
| `filename` | VARCHAR(255) | Normalized original filename |
| `content_type` | VARCHAR(100) | Validated MIME type |
| `created_at` | TIMESTAMPTZ | Database-generated creation time |
| `processed` | BOOLEAN | Backward-compatible indexed indicator |
| `status` | VARCHAR(20) | `pending`, `processing`, `indexed`, or `failed` |
| `error_message` | TEXT, nullable | Safe processing failure description |

The database check constraint `ck_documents_status` rejects unknown states.
New uploads start as `pending`; a worker claims them as `processing`, then sets
`indexed` only after PostgreSQL chunks and Qdrant vectors are persisted. Failed
work moves to `failed` and records `error_message`. Deleting a knowledge base
cascades to its documents.

## `document_chunks` table

Each row stores one ordered segment of extracted document text.

| Column | Type | Purpose |
| --- | --- | --- |
| `id` | UUID, primary key | Application-generated chunk identifier |
| `document_id` | UUID, foreign key | Parent document |
| `chunk_index` | INTEGER | Zero-based order within the document |
| `content` | TEXT | Extracted chunk text |
| `created_at` | TIMESTAMPTZ | Database-generated creation time |

The `(document_id, chunk_index)` pair is unique. `document_id` is indexed for
document-scoped reads. Its foreign key uses `ON DELETE CASCADE`, so deleting a
document removes its relational chunks. The corresponding Qdrant vectors must
still be removed through the application service because PostgreSQL cascades do
not cross storage systems.
